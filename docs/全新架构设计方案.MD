我们从第一性原理出发，重新审视“流式对话交互”与“重度后台调研”的整合，确实存在一个更优雅、更符合当前业界最佳实践的设计模式。

这个模式称为：Agent-as-a-Tool（子智能体作为工具 / 主从架构）

核心思想：不要让图(Graph)控制对话，让对话控制图
您目前的痛点在于试图让 LangGraph 统管全局，但又舍不得 chat_service.py 里手动 ReAct 循环带来的原生 Token 级流式输出和深度思考（Thinking）体验。

更好的设计是：保留 chat_service.py 作为全局唯一的入口和主控制器（Supervisor），将整个 target_discovery_graph 封装成一个常规的异步工具（Tool），挂载到主控制器的工具列表中。

全新架构设计方案
1. 意图识别的“降维打击”（回归 Function Calling）
不需要再写一个专门的 intent_router 节点来进行硬编码的规则路由。
您只需要向主 LLM 提供一个名为 run_target_discovery 的工具。当用户输入“帮我做 TDP-43 靶点分析”时，主 LLM 的原生 Function Calling 能力会自然而然地决定调用这个工具，就像它决定调用 query_pubmed 一样。

2. 将 LangGraph 降级为“重型工具” (Heavy-duty Tool)
在 app/tools/ 目录下创建一个新工具 DeepResearchTool，它的内部实现就是去触发那 5 个节点的靶点发现子图。
# 伪代码：将 LangGraph 包装为工具
class DeepResearchTool(BaseTool):
    name = "run_target_discovery"
    description = "执行深度靶点调研流水线。需要耗时几分钟。当你需要为一个靶点生成完整分析报告时调用此工具。"
    
    async def _arun(self, query: str):
        # 内部调用 LangGraph
        td_graph = build_target_discovery_graph(provider)
        final_state = await td_graph.ainvoke({"target_query": query})
        # 返回 JSON 或 Markdown 报告
        return json.dumps(final_state["final_report"])

3. 主循环的进度透传 (UX 优化)
深度调研可能耗时 3-5 分钟。在 chat_service.py 中拦截这个特定的工具调用，利用 LangGraph 的 astream 实时获取子图执行进度，并通过现有的 SSE 发给前端。

# chat_service.py 中的 _execute_tool 改进逻辑
if tc.tool_name == "run_target_discovery":
    # 告诉前端：进入深度思考模式
    yield _sse({"event": "tool_use_start", "data": "启动深度靶点分析..."})
    
    td_graph = build_target_discovery_graph(provider)
    async for output in td_graph.astream({"target_query": tc.tool_args["query"]}):
        # 监听到 graph 节点切换（如 literature -> composition）
        for node_name in output:
            yield _sse({"event": "research_progress", "data": f"正在完成阶段: {node_name}"})
            
    # 拿到最终报告，作为 ToolMessage 返回给大模型
    result = final_state["final_report"]

4. 记忆闭环的自然形成
当 run_target_discovery 工具执行完毕后，长篇的 JSON/Markdown 报告会作为标准的 ToolMessage 插入到 chat_service.py 维护的 messages 列表中。
主 LLM 拿到这个巨大的 ToolMessage 后，会生成一段友好的总结（例如：“我已经为您完成了 TDP-43 的深度调研，报告如下...”）。
最终，这条包含完整报告的 ToolMessage 和 AI 的回答，都会被现有的 append_message 存入历史记录。用户后续的追问，大模型自然能从历史上下文中读取到这份报告。

为什么这个方案远胜于 A 和 B？
保护核心资产，无需重写底层：
您的 GeminiProvider 和 DeepSeekProvider 已经能完美处理原生 SSE 甚至复杂的 thinking 过程。本方案完全不需要您去把它们硬套进 LangChain 的 BaseChatModel 接口（抛弃繁重的方案 A）。

彻底消除“大脑分裂”：
系统只有一条唯一的执行路径——前端永远调用 chat_service.py。不会再有 LangGraph 和 ReAct 争夺控制权的问题。

极佳的可扩展性：
今天您有一个 target_discovery 图。明天如果您开发了一个由多个 Agent 组成的 molecular_docking_graph（分子对接图），您只需再加一个 Tool。主循环代码一行都不用改。

解决方案 B 的致命缺陷（记忆丢失）：
方案 B 在生成报告后直接 return，导致历史上下文断裂。而“Agent-as-a-Tool”模式下，报告是作为工具返回结果（ToolMessage）存在于对话流中的，天然支持后续的“就着报告继续聊”。

需要注意的一个工程挑战：上下文窗口膨胀
这个方案唯一的挑战是：深度调研工具返回的 ToolMessage 可能极大（几千上万 token）。如果直接塞进对话历史，后续每轮对话都会携带这个巨大的包袱。

解决方案（可选结合您的现有代码）：
当 run_target_discovery 执行完毕后，不将其完整内容作为明文 ToolMessage 放入对话历史，而是将其保存为后端的 Session File (或存入 S3)。
返回给主 LLM 的 ToolMessage 只是一个索引：“报告已生成，包含靶点构成、通路、药物等 5 个维度的分析。已保存为文件 ID: 12345”。
这样，既能在当前轮次给予用户报告，又能在后续对话中利用您已经写好的 _load_file_context 机制动态加载所需内容，极大节省 Token 成本并防止大模型失焦。

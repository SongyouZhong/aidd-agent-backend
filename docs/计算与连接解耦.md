# 计算与连接解耦：AIDD Agent 长任务异步化完整方案

## 一、核心痛点分析

### 第一性原理：计算生命周期 vs. 网络生命周期

在当前架构中，AI 推理与 Tool 调用的计算任务生命周期，被死死绑定在了 HTTP 网络连接的生命周期上。

`chat_service.py` 中 `stream_chat` 的执行顺序：

1. 加载历史记录
2. 保存 User 消息
3. 进入 while 循环进行大模型推理和工具调用，通过 `yield _sse(...)` 将数据吐给前端
4. 循环结束后，执行 `await append_message(..., assistant_msg)` 将 AI 的回复存入数据库/S3

**致命缺陷：** 由于 FastAPI 的 `StreamingResponse` 特性，当用户刷新网页或网络断开时，Uvicorn 检测到连接断开后会向异步生成器 `stream_chat` 抛出 `asyncio.CancelledError`，导致整个任务立即终止，第 4 步的持久化存储代码永远无法执行。大模型已生成的文本、工具调用结果全部丢失。用户重新刷新时，前端发现少了一条回复或状态不同步，从而显示 "An error occurred."。

---

## 二、优化方案：计算、连接与交付的彻底解耦

**核心理念：**
- **计算层**：后台独立运行的工单（Ticket），不受网络波动影响
- **连接层**："状态订阅器"，随时可断，重连后瞬间同步进度
- **交互层**：主聊天框负责"交流与接单"，右侧面板（To-dos & Results）负责"进度追踪与成果交付"

---

## 三、架构设计决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 任务运行时 | 进程内 `asyncio.create_task` + Redis 状态 | 不引入 Celery/arq，通过 `BackgroundRunner` 抽象保留迁移口 |
| 任务持久化 | Redis 为权威（TTL 7d），PostgreSQL 可选审计 | 简单、无额外依赖 |
| 进度通道 | 独立 SSE `/sessions/{id}/events`，与主 `/chat` 分离 | 主流秒级结束，不被 10 分钟任务阻塞 |
| 任务取消 | 本期实现 `DELETE /api/v1/tasks/{task_id}` | 长任务场景常用 |
| 同 target 去重 | 不去重 | 每次请求独立创建新任务 |
| 用户额度/计费 | 不需要相关设计 | — |
| 前端状态库 | 引入 Zustand | 轻量，与现有 React Query 互补，组件外可调用 |

### 阶段 0：止血——刷新不再丢已生成内容（独立可上线）

**痛点直接修复，不依赖后续阶段，可立即上线。**

**后端 `app/services/chat_service.py`：**

将"保存 assistant 消息 + 写 trace"抽出为 `_finalize(...)` 函数，在生成器的 `try/finally` 中用 `asyncio.shield(_finalize(...))` 调用：

```python
async def stream_chat(...):
    assistant_text = ""
    tool_calls = []
    file_ids = []
    try:
        async for event in _run_react_loop(...):
            # 累积 assistant_text、tool_calls、file_ids
            yield event
    finally:
        # CancelledError 不阻断此处，asyncio.shield 保护持久化
        await asyncio.shield(
            _finalize(session_id, assistant_text, tool_calls, file_ids, partial=True)
        )
```

- 即使流被取消，也用已累计的内容持久化，标记 `partial=True` 供前端识别
- `partial=True` 的消息展示时可加灰色标注"（已中断）"，但内容可见

**前端 `src/component/chat/ChatContainer.tsx`：**

`onError` 回调改为先尝试从后端拉取已持久化的消息，再决定是否显示错误：

```typescript
onError: async (error) => {
  // 先 invalidate，让 React Query 重新拉取后端版本
  await queryClient.invalidateQueries({ queryKey: ['messages', projectId, sessionId] });
  // 只有在后端确实没有该消息时才显示错误占位
  const stored = queryClient.getQueryData<Message[]>(['messages', projectId, sessionId]);
  const hasStoredMsg = stored?.some(m => m.role === 'assistant');
  if (!hasStoredMsg) {
    setLocalMessages(prev => prev.map(msg =>
      msg.id === aiMessageId ? { ...msg, content: 'An error occurred.', isThinking: false } : msg
    ));
  }
  setIsStreaming(false);
}
```

**前端 `src/service/chat.ts`：**

区分错误类型，分别回调：
- `AbortError` → 用户主动取消，不显示错误
- 网络断开（fetch 失败）→ 触发 invalidate 拉取后端版本
- 服务端 `error` 事件 → 才显示错误提示

### 阶段 2：长工具非阻塞化 + 自动续轮

**新增 `app/services/background_runner.py`：**

```python
class BackgroundRunner:
    """进程内任务调度抽象。后续可替换为 arq/Celery 而不改调用方。"""
    _tasks: set[asyncio.Task] = set()

    async def submit(self, coro: Coroutine, task_id: str) -> None:
        task = asyncio.create_task(coro)
        task.set_name(task_id)
        self._tasks.add(task)
        task.add_done_callback(self._tasks.discard)

    async def cancel(self, task_id: str) -> bool:
        for t in self._tasks:
            if t.get_name() == task_id:
                t.cancel()
                return True
        return False

    async def shutdown(self):
        """在 app lifespan 关闭时调用，等待所有任务完成或取消。"""
        for t in list(self._tasks):
            t.cancel()
        await asyncio.gather(*self._tasks, return_exceptions=True)

background_runner = BackgroundRunner()
```

**`app/main.py` lifespan 适配：**

```python
@tool
async def run_target_discovery(target_query: str) -> str:
    ctx = ChatRequestContext.get()
    task_id = await task_registry.create_task(
        session_id=ctx.session_id,
        user_id=ctx.user_id,
        project_id=ctx.project_id,

        target=target_query,

        _run_discovery_background(task_id, target_query, ctx),
        task_id=task_id,
    )
    # 立即返回给 LLM，主聊天 SSE 秒级完成
    return json.dumps({
        "status": "accepted",
        "task_id": task_id,
        "target": target_query,
        "eta_seconds": 300,
        "message": "Target discovery task accepted. Running in the background — results will be appended to this conversation when ready.",
    }, ensure_ascii=False)
```

**后台协程 `_run_discovery_background`：**

1. `task_registry.update_progress(task_id, 5, "composition", "Resolving target structure...")`
2. 调用 `_run_pipeline(target_query, progress_callback=...)` —— progress_callback 直接调用 `task_registry.update_progress`
3. 报告落盘（`target_report_service.save_report_as_session_file`）
4. `task_registry.complete(task_id, result={file_ids, target, counts})`
5. 触发自动续轮：`await chat_service.resume_after_task(session_id, task_id)`

**自动续轮 `chat_service.resume_after_task`（关键闭环）：**

> 不做这一步，LLM 看不到真实结果，用户后续追问"刚才那个靶点的通路是什么"时模型会失忆。

```python
async def resume_after_task(session_id: str, task_id: str):
    """
    任务完成后由后台协程调用，不依赖任何 HTTP 连接。
    1. 把真实 tool_result 注入会话历史（作为 tool message）
    2. 调用 LLM 生成最终 assistant 总结消息并落库
    3. 广播 message_appended 事件
    """
    task = await task_registry.get(task_id)
    tool_result_msg = ToolMessage(
        tool_call_id=task.tool_call_id,
        content=json.dumps(task.result, ensure_ascii=False),
    )
    await append_message(session_id, tool_result_msg)
    messages = await load_messages(session_id)
    summary = await provider.generate(messages)  # 非流式，无需 SSE
    await append_message(session_id, AssistantMessage(content=summary))
    await task_registry.publish_event(session_id, {
        "type": "message_appended",
        "session_id": session_id,
        "message_id": summary_message_id,
    })
```

**新增 `app/api/events.py`：**

```
GET /api/v1/sessions/{session_id}/events
```

- 鉴权：`Depends(get_current_user)`，验证 session 归属

- 以 SSE 格式推送事件，每 15 秒发送 keep-alive comment
- 前端使用 fetch+ReadableStream（与 chat.ts 同模式，携带 JWT header）

```python
async def session_events_stream(session_id: str, current_user: User):
    pubsub = await get_pubsub()
    await pubsub.subscribe(f"session_events:{session_id}")
    try:
        while True:
            msg = await asyncio.wait_for(
                pubsub.get_message(ignore_subscribe_messages=True), timeout=15
            )
            if msg:
                yield f"data: {msg['data']}\n\n"
            else:
                yield ": keep-alive\n\n"
    finally:
        await pubsub.unsubscribe(f"session_events:{session_id}")
```

## 五、断线重连完整体验流程

```
用户发消息 → POST /chat
  └── LLM 调用 run_target_discovery
        └── 立即返回 accepted，主聊天 SSE 在 ~3 秒内结束（message_end）
              └── 后台任务开始，进度通过 /sessions/{id}/events 推送
                    └── 右栏出现进度条，5 个阶段逐步更新
                          └── 完成 → task_completed + message_appended 事件
                                └── 聊天区出现自动续轮的 assistant 总结消息

  └── MainChatPage 挂载
        ├── useMessages     → GET /messages（含已落库的"已受理"assistant 消息）
        ├── useActiveTasks  → GET /active-tasks（返回 running 任务，percent=65）
        │     └── taskStore.setAll(tasks)  →  右栏立刻显示 65% 进度条
        └── subscribeSessionEvents → 订阅 /sessions/{id}/events
              └── 继续接收剩余进度，直到 task_completed + message_appended
```

## 七、验收标准

1. **阶段 0 回归**：客户端中途断开后，1 秒内 `GET /messages` 能看到 `partial=true` 的 assistant 消息，内容为已生成的部分；不再出现 "An error occurred."。
2. **主流秒级完成**：`POST /chat` 触发 `run_target_discovery` 后，主聊天 SSE 在 ~3 秒内返回 `message_end`；不被 10 分钟任务阻塞。
3. **进度实时推送**：并行订阅 `/sessions/{id}/events`，收到 ≥5 个 `task_progress`、1 个 `task_completed`、1 个 `message_appended`。
4. **自动续轮落库**：`GET /messages` 最终出现自动续轮生成的 assistant 总结消息，报告文件 Chip 可点击预览。
5. **断线重连**：执行中 abort 主流 → `GET /active-tasks` 返回 running 任务 → 重新订阅 events 收剩余进度；右栏 1 秒内恢复进度条显示。
6. **进程重启**：running 任务被标记为 `failed(process_restart)` 并广播；前端右栏显示失败状态。
7. **任务取消**：`DELETE /tasks/{task_id}` 正确停止后台协程，广播 `task_cancelled`，右栏更新为已取消状态。

# 计算与连接解耦 — 实施总结

> 对应方案文档：`计算与连接解耦.md`
> 实施日期：2026-05-13

---

## 一、背景与目标

原有实现中，靶点发现（Target Discovery）等耗时计算任务直接在 SSE 流的 `async for` 循环内执行。当客户端断开连接时，FastAPI 会向生成器抛出 `asyncio.CancelledError`，导致：

1. 计算任务被强制中止，进度丢失。
2. 助手消息未能持久化到数据库，刷新后对话记录消失。
3. 前端无法感知任务状态，只能被动等待。

目标：**将"计算"与"连接"彻底解耦** — 后台任务独立运行，连接断开只影响推流，不影响计算；前端通过独立的 SSE 事件流实时感知任务进度。

---

## 二、后端改动

### 2.0 Phase 0 — 连接断开保护（`chat_service.py`）

**问题**：`stream_chat` 生成器在 `[DONE]` 之后才保存助手消息，客户端提前断开会触发 `CancelledError`，保存代码永远不会执行。

**解决方案**：
- 将消息持久化逻辑提取为独立的异步函数 `_finalize(session_id, assistant_msg_id, full_text, produced_file_ids, trace_steps)`。
- 在 `stream_chat` 生成器中，用 `try / finally` 包裹整个 SSE 推流循环，`finally` 块调用 `asyncio.shield(_finalize(...))` 确保即使外层任务被取消，持久化协程依然完整执行。

```python
try:
    # ... 推流循环 ...
    yield "data: [DONE]\n\n"
finally:
    asyncio.ensure_future(asyncio.shield(_finalize(...)))
```

**新增函数**：
- `_finalize(...)` — 保存助手消息、上传 S3 追踪文件。
- `resume_after_task(session_id, task_id)` — 后台任务完成后，重新调用 LLM 生成一条"任务完成通知"助手消息，并通过 Redis Pub/Sub 广播 `message_appended` 事件。

### 2.1 Phase 1 — Redis 扩展与任务注册表

#### `app/storage/redis_client.py`（修改）
新增两个异步辅助方法：
- `publish(channel, message)` — 向指定频道发布 JSON 消息。
- `get_pubsub()` — 返回 Redis PubSub 实例，供 SSE 端点订阅。

#### `app/services/task_registry.py`（新建）
任务状态机，以 **Redis Hash** 为持久化层：

| Redis 键 | 用途 |
|---|---|
| `task:{task_id}` | Hash，存储任务全部字段，TTL 7 天 |
| `session_tasks:{session_id}` | Set，存储该会话的所有 task_id |
| `session_events:{session_id}` | Pub/Sub 频道，推送实时事件 |

**任务状态流转**：
```
pending → running → succeeded
                 ↘ failed
                 ↘ cancelled
```

**主要函数**：

| 函数 | 说明 |
|---|---|
| `create_task(...)` | 创建任务，写入 Redis Hash，加入 session Set |
| `set_tool_call_id(task_id, tool_call_id)` | 关联 LLM tool_call_id |
| `update_progress(task_id, percent, phase, desc)` | 更新进度，发布 `task_progress` 事件 |
| `complete(task_id, result)` | 标记成功，发布 `task_completed` 事件 |
| `fail(task_id, error)` | 标记失败，发布 `task_failed` 事件 |
| `cancel(task_id)` | 标记取消，发布 `task_cancelled` 事件 |
| `get(task_id)` | 读取任务状态 |
| `list_by_session_recent(session_id, limit)` | 返回最近 N 个任务 |
| `reap_stale_tasks(session_id?)` | 清理长期处于 running 状态的僵尸任务 |

### 2.2 Phase 2 — 后台运行器与工具重构

#### `app/services/background_runner.py`（新建）
单例 `BackgroundRunner`，封装 `asyncio.create_task`：
- `submit(coro, task_id)` — 创建异步任务，注册到内部字典。
- `cancel(task_id)` — 取消指定任务。
- `shutdown()` — 优雅关闭，取消所有未完成任务（供 lifespan 调用）。

#### `app/tools/deep_research.py`（重构）
`run_target_discovery` @tool 函数改为**立即返回**：
1. 调用 `task_registry.create_task(...)` 创建任务记录。
2. 调用 `background_runner.submit(_run_discovery_background(...))` 在后台运行。
3. 立即返回 `{"status": "accepted", "task_id": ..., "target": ..., "eta_seconds": 300}`。

后台协程 `_run_discovery_background(task_id, target_query, session_id, user_id, project_id)`：
- 覆盖 `progress_callback` ContextVar，使 LangGraph 节点每次回调时向 Redis 发布进度。
- 运行 `_run_pipeline(target_query)` 执行完整的 6 节点 LangGraph 流水线。
- 保存报告 Markdown 文件到 S3。
- 调用 `task_registry.complete(task_id, result)` 标记完成。
- 懒加载调用 `chat_service.resume_after_task(session_id, task_id)` 触发 LLM 回复。

> **避免循环导入**：`chat_service` 导入了 `tools` 包，因此 `_run_discovery_background` 在函数体内部用 `from app.services.chat_service import resume_after_task` 进行懒加载。

**进度映射表**（`_PHASE_PERCENT` / `_PHASE_DESC`）：

| LangGraph 节点 | 百分比 | 描述 |
|---|---|---|
| `composition_analysis` | 10% | Analyzing composition |
| `literature_mining` | 30% | Mining literature |
| `function_analysis` | 50% | Analyzing function |
| `pathway_enrichment` | 65% | Enriching pathways |
| `drug_association` | 80% | Associating drugs |
| `synthesize` | 95% | Synthesizing report |

### 2.3 Phase 3 — 新增 API 端点

#### `app/api/events.py`（新建）
```
GET /api/v1/projects/{project_id}/sessions/{session_id}/events
```
- 验证会话归属权（调用 `session_service.get_session`）。
- 订阅 Redis Pub/Sub 频道 `session_events:{session_id}`。
- 以 `StreamingResponse` 推送 SSE，支持 `text/event-stream`。
- 每 15 秒发送 `: keep-alive` 心跳防止连接超时。

#### `app/api/tasks.py`（新建）
```
DELETE /api/v1/tasks/{task_id}
```
- 验证任务归属权（task 的 `user_id` 与当前用户匹配）。
- 若任务已处于终态（`succeeded / failed / cancelled`），拒绝请求。
- 调用 `background_runner.cancel(task_id)` 终止 asyncio Task。
- 调用 `task_registry.cancel(task_id)` 更新 Redis 状态。

#### `app/api/sessions.py`（修改）
新增：
```
GET /api/v1/projects/{project_id}/sessions/{session_id}/active-tasks
```
调用 `task_registry.list_by_session_recent`，返回最近 20 个任务的列表。

#### `app/main.py`（修改）
- Lifespan 启动时：调用 `reap_stale_tasks()` 清理上次进程留下的僵尸任务。
- Lifespan 关闭时：调用 `background_runner.shutdown()` 优雅停止所有后台任务。
- 注册新路由：`events_router.router` 和 `tasks_router.router`。

## 三、前端改动

### 3.1 Zustand 任务状态仓库（`src/store/taskStore.ts`，新建）

使用 **Zustand v5** 创建模块级单例 `useTaskStore`，存储所有活跃任务的状态：

```typescript
interface TaskState {
  task_id: string;
  kind: string;
  status: 'pending' | 'running' | 'succeeded' | 'failed' | 'cancelled';
  percent: number;
  phase: string;
  desc: string;
  target: string;
  started_at: string;
  finished_at?: string;
  result?: Record<string, unknown>;
  error?: string;
}
```

操作方法：`upsert(task)` / `remove(task_id)` / `setAll(tasks)` / `clear()`

---

### 3.2 SSE 事件服务（`src/service/events.ts`，新建）

`subscribeSessionEvents(sessionId, projectId, handlers, signal)`：
- 使用 `fetch` + `ReadableStream` 消费 SSE（支持携带 JWT Authorization 头，规避 `EventSource` 无法设置 Header 的限制）。
- 支持 `AbortSignal` 取消订阅。
- 处理事件类型：

| 事件类型 | 处理动作 |
|---|---|
| `task_progress` | 调用 `handlers.onTaskProgress(task)` |
| `task_completed` | 调用 `handlers.onTaskCompleted(task)` |
| `task_failed` | 调用 `handlers.onTaskFailed(task)` |
| `task_cancelled` | 调用 `handlers.onTaskCancelled(task)` |
| `message_appended` | 调用 `handlers.onMessageAppended(sessionId)` |

---

### 3.3 服务层扩展

**`src/service/sessions.ts`**：新增 `getActiveTasks(projectId, sessionId)` — 调用 `active-tasks` 端点，返回 `TaskState[]`。

**`src/service/queries.ts`**：新增 `useActiveTasks(projectId?, sessionId?)` — React Query hook，`staleTime: 0` 确保每次都重新拉取。

**`src/service/chat.ts`**：
- 新增 `TaskAcceptedInfo` 接口：`{ task_id: string; target: string }`。
- `tool_use_end` 事件处理：若 tool result 中 `status === 'accepted'`，则调用 `onTaskAccepted(info)`。
- `onError` 回调新增 `isAbort?: boolean` 参数：用户主动中止时传入 `true`，避免误报错误。

---

### 3.4 右侧面板（`src/component/layout/RightSection.tsx`，重写）

连接 `useTaskStore`，将任务分为两类展示：

**进行中（To-dos）**：
- `TaskProgressCard` 组件：显示目标名称、取消按钮（调用 `DELETE /tasks/{id}`）、进度描述文字、`LinearProgress` 进度条（带百分比）。

**已完成（Results）**：
- `TaskResultChips` 组件：已成功任务显示 MD 文件 Chip，可点击下载报告。

---

### 3.5 主聊天页面（`src/pages/MainChatPage.tsx`，重写）

**会话切换时**：
1. 调用 `useActiveTasks` 拉取服务端持久化的任务列表，调用 `store.setAll(tasks)` 水合 Zustand。
2. 中止旧 SSE 订阅（`AbortController.abort()`），建立新订阅（`subscribeSessionEvents(...)`）。

**SSE 事件处理**：
- `onTaskProgress / onTaskCompleted / onTaskFailed / onTaskCancelled` → `store.upsert(task)` 更新 UI。
- `onMessageAppended` → `queryClient.invalidateQueries(['messages', ...])` 拉取新消息。

---

### 3.6 聊天容器（`src/component/chat/ChatContainer.tsx`，修改）

- **`onTaskAccepted`**：工具调用返回 `accepted` 时，立即向 Zustand store 写入一条 `status: 'pending'` 的任务卡片，用户无需等待第一个 SSE 事件即可看到反馈。
- **`onError`（Phase 0 恢复逻辑）**：
  - 若 `isAbort === true`（用户主动中止），仅清空流式状态，不显示错误。
  - 否则，先调用 `queryClient.invalidateQueries` 尝试从服务端恢复持久化的助手消息；只有当确认没有存储的消息时，才在界面显示 "An error occurred."。

## 四、数据流总览

```
用户触发工具调用
    │
    ▼
run_target_discovery()
    │  立即返回 {status: "accepted", task_id}
    │
    ├─► task_registry.create_task()  →  Redis Hash
    │
    └─► background_runner.submit(_run_discovery_background)
              │
              ▼
         LangGraph Pipeline（6 节点）
              │ 每节点完成
              ▼
         task_registry.update_progress()
              │
              ▼
         Redis PUBLISH session_events:{session_id}
              │
              ▼
         events.py SSE 端点推送给前端
              │
              ▼
         subscribeSessionEvents 接收
              │
              ▼
         useTaskStore.upsert()  →  RightSection 更新 UI

         完成后：
         task_registry.complete()
         chat_service.resume_after_task()
              │
              ▼
         LLM 生成通知消息 → 持久化 → publish message_appended
              │
              ▼
         前端 invalidateQueries → 消息列表刷新
```

## 五、关键设计决策

| 决策 | 原因 |
|---|---|
| 用 Redis Hash 存储任务状态而非内存 | 进程重启后状态可恢复，支持多实例部署 |
| 用 Redis Pub/Sub 而非 WebSocket | 复用已有 Redis，无需引入新依赖；SSE 对单向推送已足够 |
| `asyncio.shield` 保护 `_finalize` | `CancelledError` 会取消外层 Task，shield 使内层协程继续执行 |
| 懒加载 `chat_service` | 避免 `deep_research` ↔ `chat_service` 循环导入 |
| Zustand 替代 React Query 存储任务状态 | 任务状态来源是 SSE，不是 REST 查询；Zustand 更适合事件驱动更新 |
| fetch + ReadableStream 替代 EventSource | `EventSource` 不支持自定义 Header，无法携带 JWT |

# 架构偏差分析与修复方案

> 文档目的：记录设计文档与实际实现之间的偏差，分析根本原因，并提供两条修复路径的详细方案。

---

## 1. 问题现象

用户通过前端聊天界面（`POST /api/v1/chat`）发送靶点分析请求（如"帮我做 TDP-43 靶点分析"），后端日志仅显示：

```
INFO: "POST /api/v1/chat HTTP/1.1" 200 OK
```

没有任何工具调用日志（PubMed、UniProt、KEGG、ChEMBL 等）。而同一问题通过 `langgraph dev` 路径执行时，日志中出现大量工具调用，并触发完整的 5 节点并行靶点发现流水线。

---

## 2. 设计预期 vs 实际实现

### 2.1 设计文档预期（`后端设计文档.md` §6.3）

设计文档中，`chat_service.stream_chat()` 应该这样实现：

```python
# 设计文档中的预期实现
provider = get_default_provider()
agent = build_agent(provider)  # ← 使用 LangGraph 图
messages = _history_to_langchain(history)
messages.append(HumanMessage(content=user_content))

# 使用 LangGraph 的 astream_events 获取流式事件
async for event in agent.astream_events(
    {"messages": messages, "hot_loaded": set()},
    version="v2",
):
    kind = event["event"]
    if kind == "on_chat_model_stream":      # token 级流式
        ...
    elif kind == "on_tool_start":           # 工具调用开始
        ...
    elif kind == "on_tool_end":             # 工具调用结束
        ...
```

调用链：

```
POST /api/v1/chat
  └─ chat_service.stream_chat()
       └─ agent.astream_events()        ← LangGraph 图
            ├─ intent_router_node        ← 意图识别
            ├─ target_discovery_node     ← 靶点发现子图（5节点并行）
            │    ├─ literature_node
            │    ├─ composition_node
            │    ├─ function_node
            │    ├─ pathway_node
            │    └─ drugs_node + synthesize_node
            └─ agent_node (通用 ReAct)
```

### 2.2 实际实现（`app/services/chat_service.py`）

实际代码完全绕过了 LangGraph 图，手动实现了一个 ReAct 循环：

```python
# 实际代码
provider = get_default_provider()
# 注意：没有 build_agent()，没有图，没有 astream_events

while total_rounds < MAX_TOOL_ROUNDS:
    async for chunk in provider.stream(llm_messages, tools=tools_for_llm):
        # 直接调用 provider.stream()，完全绕过 LangGraph 图
        ...
```

调用链：

```
POST /api/v1/chat
  └─ chat_service.stream_chat()
       └─ provider.stream()             ← 直接调用 LLM，绕过图
            └─ LLM 回答（可能不调工具）
```

### 2.3 对比表

| 功能 | 设计预期 | 实际实现 |
|---|---|---|
| **执行引擎** | `agent.astream_events()` (LangGraph 图) | 手动 `while` ReAct 循环 |
| **意图路由** | ✅ `intent_router_node` 强制分派 | ❌ 无，LLM 自行决定是否调工具 |
| **靶点发现子图** | ✅ 5 节点并行流水线 | ❌ 无 |
| **工具热加载** | ✅ 图自动管理 | ⚠️ 手动维护 `hot_loaded` 集合 |
| **Token 流式** | ✅ 工作（通过 `on_chat_model_stream`） | ✅ 工作（通过 `provider.stream()`） |
| **LangGraph Checkpoint** | ✅ 自动状态持久化 | ❌ 无 |
| **Auto-Compaction** | ✅ `context_manager.py` 集成 | ⚠️ 代码存在但未被调用 |

## 3. 根本原因分析

### 3.1 技术障碍：Provider 不是 LangChain ChatModel

`astream_events` 的 `on_chat_model_stream` 事件只有在使用 **LangChain 原生 ChatModel**（如 `ChatGoogleGenerativeAI`、`ChatOpenAI`）时才会触发。

本项目的 LLM Provider 实现绕过了 LangChain 的 ChatModel 接口，直接使用原生 SDK：

| Provider | 使用方式 | 是否 LangChain ChatModel |
|---|---|---|
| `GeminiProvider` | 直接调用 `google.genai.Client` | ❌ |
| `DeepSeekProvider` | 直接调用 `openai.AsyncOpenAI` | ❌ |
| `QwenProvider` | 直接调用 `openai.AsyncOpenAI` | ❌ |

这是绕过原因：当时尝试 `astream_events` 时，无法收到 token 级别的 delta，导致被迫放弃图驱动方案，改为手动 ReAct 循环直接调用 `provider.stream()`。

### 3.2 核心矛盾

```
设计目标：用 LangGraph 图驱动执行（intent 路由 + 子图 + checkpoint）
技术现实：Provider 不实现 LangChain 接口，astream_events 无法返回 token 流
妥协结果：手动 ReAct 循环，直接 provider.stream()，完全绕过图
副作用：intent_router、target_discovery 子图、checkpoint 全部失效
```

### 3.3 两条代码路径的形成

| 路径 | 入口 | 调用链 |
|---|---|---|
| **前端聊天** | `POST /api/v1/chat` → `chat_service.py` | 手动 ReAct，`provider.stream()` |
| **LangGraph dev** | `langgraph dev` → `graph.py:graph` | LangGraph 图，`provider.generate()` |

这两条路径使用完全相同的 `provider` 和 `tools`，但执行逻辑完全独立，互不影响。`langgraph dev` 是 LangGraph Studio 的调试入口，**不是**前端聊天的执行路径。

## 4. 修复方案

### 方案 A：改造 Provider，实现 LangChain ChatModel 接口（推荐，架构干净）

**思路**：让每个 Provider 包装为一个继承自 `BaseChatModel` 的 LangChain 对象，实现 `_stream()` 和 `_generate()` 方法。这样 `astream_events` 就能正确触发 `on_chat_model_stream` 事件，`chat_service.py` 就能改回设计文档中的 `agent.astream_events()` 调用方式。

**改动范围**：
- `app/agent/llm_provider.py`：为每个 Provider 增加 `BaseChatModel` 包装类
- `app/services/chat_service.py`：改为 `agent.astream_events()` 调用
- `app/agent/agent.py`：`agent_node` 改为调用包装后的 ChatModel

**改动示意**：

```python
# llm_provider.py — 新增 LangChain 包装
from langchain_core.language_models.chat_models import BaseChatModel
from langchain_core.messages import BaseMessage as LCMessage
from langchain_core.outputs import ChatGeneration, ChatGenerationChunk, ChatResult

class GeminiChatModel(BaseChatModel):
    """LangChain-compatible wrapper for GeminiProvider."""
    _provider: GeminiProvider

    def _generate(self, messages, stop=None, **kwargs) -> ChatResult:
        resp = asyncio.get_event_loop().run_until_complete(
            self._provider.generate(messages)
        )
        return ChatResult(generations=[ChatGeneration(message=AIMessage(content=resp.text))])

    def _stream(self, messages, stop=None, **kwargs):
        async def _inner():
            async for chunk in self._provider.stream(messages):
                if chunk.type == "text":
                    yield ChatGenerationChunk(message=AIMessageChunk(content=chunk.content))
        # 将 async generator 转为同步 generator
        ...

    @property
    def _llm_type(self) -> str:
        return "gemini"
```

```python
# chat_service.py — 改回设计文档预期实现
provider = get_default_chat_model()   # 返回 BaseChatModel 包装
agent = build_agent(provider)

async for event in agent.astream_events(
    {"messages": messages, "hot_loaded": set(), "session_id": session_id},
    version="v2",
):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        chunk = event["data"].get("chunk")
        if chunk and chunk.content:
            yield _sse({"event": "content_delta", "data": {"delta": chunk.content}})
    elif kind == "on_tool_start":
        yield _sse({"event": "tool_use_start", "data": {"tool_name": event["name"]}})
    elif kind == "on_tool_end":
        output = str(event["data"].get("output", ""))
        yield _sse({"event": "tool_use_end", "data": {"result_summary": output[:200]}})
```

**优点**：
- 完全按设计文档走，架构最干净
- intent 路由、靶点发现子图、checkpoint、Auto-Compaction 全部自动生效
- 只维护一套执行逻辑

**缺点**：
- 改动量较大，需要为每个 Provider 实现 LangChain 接口
- thinking/reasoning_content 的处理需要特殊适配
- `DeepSeek` 的 `reasoning_content` 回传逻辑需要在流式接口中重新处理

---

### 方案 B：在手动 ReAct 循环中加入意图路由（快速解决，维护两套逻辑）

**思路**：保留现有手动 ReAct 循环，在第一轮 LLM 调用之前加入意图识别逻辑，检测到靶点分析意图时直接调用 `build_target_discovery_graph`，并将报告以 SSE 流式返回。

**改动范围**：
- `app/services/chat_service.py`：在 ReAct 循环开始前插入意图路由

**改动示意**：

```python
# chat_service.py — 在 ReAct 循环开始前加入意图路由
from app.agent.agent import build_agent
from app.agent.target_discovery_graph import build_target_discovery_graph

async def stream_chat(...):
    # ... 现有代码（加载历史、保存用户消息、构建 provider）...

    # ★ 新增：意图路由
    intent = await _detect_intent(provider, user_content)

    if intent.get("route") == "target_discovery":
        # 走靶点发现子图
        async for event in _stream_target_discovery(
            provider, intent["target_query"], session_id, assistant_msg_id
        ):
            yield event
        return

    # 原有手动 ReAct 循环（通用问答）
    while total_rounds < MAX_TOOL_ROUNDS:
        ...


async def _detect_intent(provider, user_content: str) -> dict:
    """快速意图分类，复用 agent.py 中的 INTENT_ROUTER_PROMPT。"""
    from app.agent.prompts.target_discovery import INTENT_ROUTER_PROMPT
    resp = await provider.generate(
        messages=[
            SystemMessage(content=INTENT_ROUTER_PROMPT),
            HumanMessage(content=user_content),
        ],
        tools=None,
        max_tokens=128,  # 只需要返回 JSON，token 消耗极少
    )
    try:
        m = re.search(r"\{.*?\}", resp.text, re.DOTALL)
        if m:
            return json.loads(m.group())
    except Exception:
        pass
    return {"route": "agent"}


async def _stream_target_discovery(provider, query: str, session_id: str, msg_id: str):
    """执行靶点发现子图并将进度以 SSE 流式发送给前端。"""
    from app.agent.target_discovery_graph import build_target_discovery_graph

    yield _sse({"event": "tool_use_start", "data": {
        "tool_name": "target_discovery",
        "tool_call_id": "td-pipeline",
        "args": {"query": query},
    }})

    td_graph = build_target_discovery_graph(provider)
    final_state = await td_graph.ainvoke({
        "target_query": query,
        "messages": [],
        "sub_results": {},
        "notes": [],
        "final_report": {},
    })

    report = final_state.get("final_report") or {}
    report_text = (
        f"## 靶点发现报告：{query}\n\n"
        f"```json\n{json.dumps(report, ensure_ascii=False, indent=2)}\n```"
    )

    yield _sse({"event": "tool_use_end", "data": {
        "tool_call_id": "td-pipeline",
        "result_summary": f"靶点发现完成，报告含 {len(report)} 个字段",
    }})

    # 逐段流式输出报告
    for line in report_text.split("\n"):
        yield _sse({"event": "content_delta", "data": {"delta": line + "\n"}})
```

**优点**：
- 改动量小，风险低
- 不影响现有流式 SSE 逻辑
- 可快速解决靶点发现问题

**缺点**：
- 维护两套逻辑（手动 ReAct + 子图），与 `agent.py` 中的 `intent_router_node` 重复
- LangGraph checkpoint、Auto-Compaction 仍然缺失
- 只解决靶点发现问题，其他需要图驱动的功能仍然缺失

## 5. 方案比较与建议

| 维度 | 方案 A（改造 Provider） | 方案 B（插入意图路由） |
|---|---|---|
| **改动量** | 大（~300 行） | 小（~80 行） |
| **架构一致性** | ✅ 与设计文档完全一致 | ⚠️ 两套逻辑并存 |
| **靶点发现修复** | ✅ | ✅ |
| **Intent 路由** | ✅ 自动（图驱动） | ✅ 手动插入 |
| **Checkpoint** | ✅ LangGraph 自动 | ❌ |
| **Auto-Compaction** | ✅ | ❌ |
| **Token 流式** | ✅ | ✅ |
| **风险** | 中（需测试所有 Provider） | 低（改动局限在 chat_service） |
| **推荐场景** | 长期维护、功能完整性优先 | 快速上线、短期解决靶点问题 |

**建议**：

- **短期**：先实施方案 B，快速修复靶点发现功能，让产品可用。
- **中期**：实施方案 A，将 Provider 包装为 LangChain ChatModel，彻底统一执行路径，消除两套逻辑。
